# 金字塔 · 横向归纳逻辑

> 金字塔横向维度之一:**从多个具体信息里归纳出一个统领结论**。像**收纳师**——先把散落的东西分门别类,再从类别里看出规律。

## 核心

**归纳 = 由下而上。** 手上有一堆碎片信息,先分类,再看出共性,最后提炼塔尖。

## WHAT:归纳的三种排列方式

同一层的论据,按什么顺序排、按什么维度分?归纳给你**三种基本排法 + 两种应用场景**。

| 归纳方式 | 说明 | 举例 |
|---------|------|------|
| **时间逻辑** | 按时间先后排列 | 第一步 → 第二步 → 第三步;Q1 → Q2 → Q3 |
| **空间逻辑** | 按空间/结构排列 | 北区 → 东区 → 南区 → 西区;市场 → 产品 → 组织 |
| **重要性逻辑** | 按重要程度排列,**最重要的放第一个** | 核心原因 → 次要原因 → 补充原因 |

### 两种常用应用场景

- **比较利弊**:方案 A 的利 / 弊 vs 方案 B 的利 / 弊 → 综合比较 → 结论(适合面对「两个选项选哪个」的场景)
- **决胜点意识**:不是所有论据权重一样,**找那个能「一击命中」的点**先放出来——这就是「重要性逻辑」的本质

## WHY:归纳为什么适合「先信息后结论」

**当你还不知道结论是什么**的时候,演绎推不出来(没有前提),只能归纳:

- 做分析、调研、例会汇报:**手上只有数据/素材**,结论需要从里面长出来
- 做项目复盘:一堆事情已经发生,要归纳出规律
- 做经验沉淀:把过去的 N 个案例抽象成方法论

归纳的关键是**分类维度选对**。同一批信息,按不同维度归纳,能得出完全不同的结论——这也是为什么归纳必须配 [[MECE原则|MECE]] 一起用:**每一层只用一个分类标准**。

## HOW:归纳三步法(自下而上搭金字塔)

1. **列出所有要点**(不做判断,先摊开)
2. **找逻辑关系**:按时间/空间/重要性归类分组
3. **从组里提炼塔尖结论**:每一组在说什么?所有组合起来在说什么?

## 示例:例会工作汇报

手上有这些碎片:

- 市场 A 本周签 2 单
- 市场 B 本周客诉多
- 市场 C 新签的客户迟迟不付款
- 市场 A 团队士气高
- 市场 B 在做售后流程改造

**按时间归纳**:本周我们做了什么?——不合适,信息分布不在时间维度。

**按空间归纳**(市场 A / B / C):

- 市场 A:2 单 + 士气高 → **势头好**
- 市场 B:客诉多 + 在做流程改造 → **正在解决问题**
- 市场 C:回款慢 → **风险点**

塔尖结论:**本周市场侧整体推进中,A 在加速、B 在修复、C 需要关注回款**。

## 归纳 vs 演绎 · 什么时候用哪个

| | 归纳 | [[金字塔-横向演绎逻辑|演绎]] |
|---|---|---|
| 方向 | 由下而上 | 由前到后(推理链) |
| 适合 | 总结大量信息 | 论证某件事的合理性 |
| 起点 | 一堆事实 | 一个大前提 |
| 风险 | 分类维度选错会偏 | 前提不成立全盘错 |

**实际应用**:一次汇报里经常**两个都用**——归纳出结论,再用演绎支撑结论。

## 检查:这组归纳站得住吗?

每一层过一遍 [[MECE原则|MECE]]:

- **互斥**:我的分类之间重不重叠?(市场 A 和「士气」不是同一维度)
- **穷尽**:合起来覆盖了全部吗?(只说了 A / B / C,D 在哪?)
